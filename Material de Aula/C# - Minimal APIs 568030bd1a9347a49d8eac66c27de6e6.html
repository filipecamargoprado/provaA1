C# - Minimal APIs

Ambiente Web

1. Criando um Projeto de API Mínima via CLI

Para criar um projeto de API mínima usando o CLI do .NET, siga estas etapas:

  • Abra o terminal ou prompt de comando.
  • Execute o seguinte comando para criar um novo projeto de API mínima:Isso criará um novo diretório chamado “MinimalApiProject” com a estrutura básica de uma API mínima.
dotnet new web -n MinimalApiProject

Isso criará um novo diretório chamado “MinimalApiProject” com a estrutura básica de uma API mínima.

2. Estrutura Básica de uma API Mínima

A estrutura básica de uma API mínima consiste em um arquivo Program.cs e um arquivo Startup.cs. Vamos focar no Program.cs:

using Microsoft.AspNetCore.Builder;
using Microsoft.Extensions.Hosting;

var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.MapGet("/", () => "Hello, Minimal API!");

app.Run();

Neste exemplo, criamos um endpoint que responde com a mensagem “Hello, Minimal API!” quando acessamos a raiz (“/”) da API.

3. Criando Endpoints

Para criar endpoints, utilize os métodos MapGet, MapPost, MapPut, etc. Por exemplo:

app.MapGet("/items", () => "Obter todos os itens");
app.MapPost("/items", () => "Adicionar um novo item");
// ... outros endpoints

4. Recebendo Parâmetros

Pela URL (Query String)

Declare um método para aceitar requisições HTTP GET com parâmetros:

app.MapGet("/items/{id}", (int id) => $"Obter item com ID {id}");

Chame o endpoint: /items/42

Pelo Corpo (JSON)

Declare um método para aceitar requisições HTTP POST com um objeto no corpo:

app.MapPost("/items", ([FromBody] TodoItem item) => $"Adicionar item: {item.Name}");

Envie um JSON no corpo da solicitação.

5. Especificando Métodos HTTP e Códigos de Retorno

Métodos HTTP (HTTP Verbs)

Os métodos HTTP desempenham um papel fundamental na comunicação entre clientes e servidores na World Wide Web. Eles definem as operações que podem ser realizadas em recursos web. Aqui estão os principais métodos:

  1. GET:
    • Solicita a representação de um recurso específico.
    • Retorna apenas dados.
    • Exemplo: Obter informações de um usuário pelo ID.
  1. POST:
    • Submete uma entidade a um recurso específico.
    • Pode causar mudanças no estado do recurso ou efeitos colaterais no servidor.
    • Exemplo: Enviar dados de um formulário para criar um novo usuário.
  1. PUT:
    • Substitui todas as representações atuais do recurso pelo conteúdo da requisição.
    • Atualiza um recurso existente.
    • Exemplo: Atualizar informações de um usuário.
  1. DELETE:
    • Remove um recurso específico.
    • Exemplo: Excluir um usuário.
  1. PATCH:
    • Aplica modificações parciais em um recurso.
    • Exemplo: Atualizar apenas alguns campos de um usuário.

    6. Códigos de Status HTTP mais comuns

    Os códigos de status HTTP são respostas de três dígitos que um servidor retorna em resposta a uma solicitação de um cliente. Esses códigos indicam o resultado da operação e são agrupados em cinco classes:

    1. Respostas Informativas (100 – 199):
      • Essas respostas são informativas e não indicam sucesso ou falha. Alguns exemplos incluem:
        • 100 Continue: Indica que o cliente deve continuar a solicitação ou ignorar a resposta se a solicitação já estiver concluída.
        • 101 Switching Protocols: Enviado quando o servidor está mudando de protocolo (por exemplo, HTTP para WebSocket).
    1. Respostas Bem-Sucedidas (200 – 299):
      • Indicam que a solicitação foi bem-sucedida. Alguns exemplos:
        • 200 OK: A solicitação foi bem-sucedida. O significado varia com o método HTTP usado (GET, HEAD, PUT, POST, TRACE).
        • 201 Created: A requisição foi bem-sucedida, e um novo recurso foi criado como resultado (geralmente após solicitações POST ou PUT).
    1. Mensagens de Redirecionamento (300 – 399):
      • Indicam que o cliente deve tomar ações adicionais para concluir a solicitação. Exemplos:
        • 301 Moved Permanently: O recurso foi movido permanentemente para uma nova URL.
        • 302 Found (ou 303 See Other): O recurso foi temporariamente movido para uma nova URL.
    1. Respostas de Erro do Cliente (400 – 499):
      • Indicam que houve um erro na solicitação do cliente. Alguns exemplos:
        • 404 Not Found: O recurso solicitado não foi encontrado.
        • 400 Bad Request: A solicitação é inválida ou malformada.
    1. Respostas de Erro do Servidor (500 – 599):
      • Indicam que o servidor encontrou um erro ao processar a solicitação. Exemplos:
        • 500 Internal Server Error: Erro interno no servidor.
        • 503 Service Unavailable: O servidor não está disponível para lidar com a solicitação.

    Lembre-se de que esses códigos são essenciais para depurar problemas e entender o estado das solicitações. Adaptar esses conceitos ao seu cenário específico é fundamental! 😊
